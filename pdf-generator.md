TRD: ReportLab JSON PDF Wrapper

1. Objective

Build a Python-based PDF generation wrapper over ReportLab that accepts a validated JSON document specification and produces polished PDFs without requiring the AI agent to generate ReportLab code.

The wrapper must support multiple report types, reusable templates, reusable components, text styling, images, tables, charts, KPIs, sections, page breaks, headers, footers, and future extensibility.

Primary goal:

LLM / Agent → compact JSON → Python renderer → ReportLab PDF

The agent should never generate Python layout code.

⸻

2. Problem Statement

Current PDF generation uses direct ReportLab Python code generated by the agent. This causes:

* High token usage
* Inconsistent layout
* Hard-to-debug generated code
* Security risk from generated executable code
* Poor reusability across report types
* Difficult maintenance of styles, branding, charts, and templates

The new system should reduce token usage by moving layout logic into backend components.

⸻

3. Goals

Functional Goals

* Accept JSON input and generate PDF.
* Support multiple report templates.
* Support reusable components.
* Support text, rich text, tables, images, charts, KPI cards, callouts, insight lists, recommendations, and section headers.
* Support page header/footer.
* Support brand themes.
* Support custom report types.
* Support local images and generated chart images.
* Validate JSON before rendering.
* Return generated PDF path, bytes, or stream.
* Allow CLI and MCP/server usage.

Non-Functional Goals

* Deterministic output.
* Safe execution, no dynamic Python code from LLM.
* Token-efficient.
* Easy to extend.
* Backend-owned styling.
* Good default design.
* Compatible with Kubernetes/server execution.
* Clear error messages.
* Testable with golden PDFs or snapshot metadata.

⸻

4. Non-Goals

* Not a full HTML/CSS renderer.
* Not a replacement for all ReportLab APIs.
* Not allowing arbitrary Python execution.
* Not letting agent control low-level coordinates.
* Not trying to support every PDF layout in v1.
* Not allowing untrusted external image access without validation.

⸻

12. Renderer API

Python API

from reportlab_json_renderer import render_pdf
result = render_pdf(
    spec=input_json,
    output_path="/tmp/report.pdf"
)
print(result.path)

Result Object

{
    "success": true,
    "path": "/tmp/report.pdf",
    "pages": 5,
    "warnings": [],
    "metadata": {
        "template": "analytics_report_v1",
        "theme": "green"
    }
}

CLI

pdf-renderer render \
  --input report.json \
  --output report.pdf

An empty `blocks` array is valid. It should still produce a single-page PDF so
callers do not get a zero-page success result.

Validate only:

pdf-renderer validate --input report.json

Rendering should fail closed on block-level errors by default. Partial output is
an explicit opt-in for host applications, not the default library behavior.

List templates:

pdf-renderer templates

List blocks:

pdf-renderer blocks

Render sample:

pdf-renderer sample --template analytics_report_v1 --output sample.pdf

14. Validation Rules

Use Pydantic or JSON Schema.

Validation should check:

* Required fields.
* Unknown block types.
* Invalid tones.
* Invalid chart types.
* Table rows matching columns.
* Column widths sum to approximately 1.0.
* Image source allowed.
* Text length limits.
* Maximum block count.
* Maximum table rows unless pagination enabled.
* Maximum image size.
* Valid page size.
* Valid template.
* Valid theme.

Example validation error:

{
  "valid": false,
  "errors": [
    {
      "path": "blocks[3].rows[2]",
      "message": "Row has 4 cells but table has 5 columns"
    }
  ]
}

⸻

15. Security Requirements

* Never execute generated Python.
* Do not allow arbitrary file paths unless whitelisted.
* Restrict image access to allowed schemes.
* Validate HTTP image URLs if enabled.
* Limit file size.
* Limit image dimensions.
* Sanitize text for ReportLab paragraph markup.
* Escape unsafe XML/HTML characters.
* Restrict fonts to registered fonts.
* Set timeout for remote downloads.
* Do not allow shell execution from JSON.
* Reject unknown block types unless explicitly enabled.

⸻

local:/safe/path/image.png

base64

17. Chart Handling

Charts should be generated by backend, not by the agent.

The agent provides:

{
  "chart_type": "bar",
  "labels": [],
  "values": []
}

Backend owns:

* Chart size
* Colors
* Fonts
* Grid
* Labels
* Legends
* Axis formatting
* DPI

For large datasets, prefer:

{
  "data_ref": "orders_by_outlet_this_week"
}

The backend resolves data_ref from precomputed data.

Skill: PDF Report JSON Generator

Purpose

Use this skill when the user asks to create a PDF report using the ReportLab JSON wrapper.

The agent must output only a valid JSON report specification. It must not generate Python, ReportLab, matplotlib, HTML, CSS, or shell commands unless explicitly asked by the user.

⸻

Core Rule

Always produce:

compact JSON spec → wrapper generates PDF

Never produce:

ReportLab code

⸻

When to Use

Use this skill for:

* Business reports
* Analytics reports
* Weekly performance reports
* Operational reports
* Invoices
* Proposals
* KPI summaries
* Merchant health reports
* Review reports
* Marketing reports
* Data-heavy PDF documents

Output Contract

The output must be valid JSON with this root structure:

{
  "version": "1.0",
  "template": "analytics_report_v1",
  "theme": "green",
  "metadata": {},
  "blocks": []
}

* Keep JSON compact.
* Do not include layout dimensions unless necessary.
* Prefer tone over color.
* Prefer data_ref if backend already has data.
* Avoid long styling instructions.
* Avoid raw ReportLab markup.
* Escape quotes correctly.
* Keep table rows aligned with table columns.
* Do not invent data.
* Use short but meaningful titles.

⸻

Good Example

{
  "version": "1.0",
  "template": "analytics_report_v1",
  "theme": "green",
  "metadata": {
    "entity_name": "Demo Store",
    "report_title": "Order Performance Evaluation Report",
    "period": "11 Jun – 17 Jun 2026"
  },
  "blocks": [
    {
      "type": "kpi_grid",
      "title": "Brand KPI Summary",
      "columns": 5,
      "items": [
        {"label": "Total Orders", "value": "3,743", "sub": "This Week", "tone": "primary"},
        {"label": "Net Sales", "value": "₹19,62,261", "sub": "↓ −12% vs LW", "tone": "danger"}
      ]
    },
    {
      "type": "callout",
      "tone": "danger",
      "title": "Brand-Wide Decline",
      "text": "Order volume is down 12.3% week over week."
    }
  ]
}

⸻

Bad Example

Do not output this:

from reportlab.platypus import Table, Paragraph
story.append(Table(...))

Reason: the wrapper owns layout and rendering.

⸻

Recommended Report Flow

For analytics reports, use this structure:

title metadata
kpi_grid
overall callouts
action items
performance rating table
source mix table/chart
outlet comparison table/chart
deep-dive sections
promo/driver analysis
trend table/chart
final recommendations
disclaimer

⸻

Handling Missing Data

If data is missing, use one of:

{
  "type": "callout",
  "tone": "warning",
  "title": "Data Missing",
  "text": "Outlet-level data was not available for this period."
}

or omit the section.

Do not fabricate numbers.

⸻

Token Efficiency Rules

Prefer this:

{
  "type": "chart",
  "chart_type": "bar",
  "data_ref": "orders_by_outlet"
}

Instead of this:

{
  "type": "chart",
  "chart_type": "bar",
  "labels": ["Outlet 1", "Outlet 2"],
  "values": [100, 200]
}

Use full values only when no data_ref exists.

⸻

Final Output Rule

When asked to generate a PDF spec, output JSON only.

When asked to explain the approach, explain normally.

When asked to create or modify the wrapper, provide implementation details, not PDF JSON.
